# 05. 메일 / 알림 서비스 (Mail / Notification Service)

## 서비스 개요

| 항목 | 내용 |
|------|------|
| 책임 | 메일 생성·발송, 발송 상태 추적, 실패 재시도 큐, 템플릿 렌더링 |
| 주요 액터 | 시스템(다른 서비스가 호출), 관리자(수동 재발송/템플릿 관리) |
| UI | 없음(발송) / 상태 표시·템플릿은 관리자 화면([03](./03-관리자-서비스.md)) |
| 핵심 엔티티 | `mail_message`(발송 큐/로그), `mail_template` |
| 외부 의존 | SMTP/메일 게이트웨이 |
| 핵심 원칙 | 비동기 발송, 멱등(중복 방지 키), 재시도+백오프, 실패 가시성 |

## 1. 메일 종류 및 발송 시점

| 코드 | 메일 종류 | 발송 시점 | 트리거 액션 | 수신자 | 주요 내용 |
|------|-----------|-----------|-------------|--------|-----------|
| `EMAIL_VERIFY` | 이메일 인증 | 회원가입 직후 / 재발송 | [ACC-P1-A2]·[ACC-P2-A2] | 가입자 | 인증 링크(24h 만료) |
| `RESERVE_DONE_NORMAL` | 예약 완료(일반) | 관리자 확정 시 | [ADM-P5-A1/A2] | 확정자 | 확정 날짜·시간 |
| `RESERVE_DONE_REAPPLY` | 예약 완료(재신청) | 재신청 즉시 확정 시 | [RSV-P3-A2] | 재신청 확정자 | 확정 날짜·시간, **취소 불가** 안내 |
| `DROP_REAPPLY_NOTICE` | 탈락/재신청 안내 | 마감(수 17:00) 자동 / 관리자 수동 | [BAT-J2-A5]·[ADM-P6-A3] | 우선권 탈락자 | 빈 슬롯, 재신청 시간(다음날 17:00까지), 선착순·즉시 확정·취소 불가 |

## 2. 발송 처리 모델

### 2.1 처리 흐름(생성 → 발송 → 재시도)
```mermaid
flowchart LR
    Caller["호출 서비스\n(계정/예약/관리자/배치)"] -->|enqueue| Q[(mail_message\nstatus=대기)]
    Q --> W[발송 워커]
    W -->|성공| OK[status=성공\nsent_at]
    W -->|실패| F[status=실패\nretry_count++]
    F --> R{retry_count < N?}
    R -- 예 --> Q
    R -- 아니오 --> D[status=실패(영구)\n대시보드 노출]
    Admin["관리자 수동 재발송"] --> Q
```

### 2.2 상태(mail_message.status)
| 상태 | 의미 | 다음 |
|------|------|------|
| 대기(PENDING) | enqueue됨, 발송 전 | → 성공/실패 |
| 발송중(SENDING) | 워커 처리 중(락) | → 성공/실패 |
| 성공(SENT) | 발송 성공 | (종료) |
| 실패(FAILED) | 일시 실패, 재시도 대상 | → 대기(재시도) |
| 실패-영구(DEAD) | 최대 재시도 초과 | 관리자 수동 재발송만 |

## 3. 액션 명세

### [MAIL-A1] 메일 enqueue (생성)
- 트리거: 타 서비스의 발송 요청
- 입력: `type, toMemberId/toEmail, context(payload), dedupeKey?`
- 처리 로직:
  1. **중복 방지 키(dedupeKey)** 검사: 동일 키 발송 이력 존재 시 생략(멱등).
     - 예: 인증=`verify:{tokenId}`, 완료(일반)=`done:{reservationId}`, 안내=`drop:{cycleId}:{memberId}`.
  2. 템플릿 조회([MAIL-A4]) 및 컨텍스트 바인딩으로 제목/본문 렌더.
  3. `mail_message` insert(`status=대기`, `retry_count=0`).
- 결과/후처리: 비동기 발송 대기.
- 관련 데이터: `mail_message`(insert), `mail_template`(read)

### [MAIL-A2] 메일 발송 (워커)
- 트리거: 발송 워커 폴링/이벤트
- 처리 로직:
  1. `대기` 또는 재시도 대상 1건 선점(`SENDING` 전이 + 락, 중복 처리 방지).
  2. SMTP/게이트웨이로 발송.
  3. 성공 → `성공`,`sent_at=now`. 실패 → `실패`,`retry_count++`,`last_error`,`last_tried_at`.
- 예외: 게이트웨이 오류/타임아웃 → 실패 처리(재시도 대상).
- 관련 데이터: `mail_message`(update)

### [MAIL-A3] 재시도 (배치)
- 트리거: [BAT-J4-A1] 주기 잡
- 처리 로직: `실패` & `retry_count<N` & 백오프 경과 건을 `대기`로 환원해 재발송. 초과 시 `실패-영구(DEAD)`.
- 백오프(예): 1분 → 5분 → 30분 (지수/고정 혼합, 운영값).
- 관련 데이터: `mail_message`(update)

### [MAIL-A4] 템플릿 렌더링/관리
- 트리거: enqueue 시 렌더 / 관리자 [ADM-P8-A3] 편집
- 처리 로직: 메일 종류별 `mail_template`(제목/본문, 치환 변수) 조회·치환. 미설정 시 기본 템플릿 사용.
- 치환 변수(예): `{name}`, `{verifyUrl}`, `{slotDate}`, `{slotTime}`, `{emptySlots}`, `{reapplyDeadline}`.
- 관련 데이터: `mail_template`(read/update)

### [MAIL-A5] 수동 재발송 (관리자)
- 트리거: 대시보드 발송 상태에서 [재발송]([ADM-P2-A4])
- 처리 로직: 대상 `mail_message`를 `대기`로 환원(또는 신규 생성)해 재발송. 권한 검증.
- 관련 데이터: `mail_message`(update/insert)

## 4. 멱등성 & 중복 방지 키 정의

| 메일 종류 | dedupeKey(예) | 의미 |
|-----------|---------------|------|
| `EMAIL_VERIFY` | `verify:{tokenId}` | 토큰 1건당 1메일 |
| `RESERVE_DONE_NORMAL` | `done:n:{reservationId}` | 확정 1건당 1메일 |
| `RESERVE_DONE_REAPPLY` | `done:r:{reservationId}` | 재신청 확정 1건당 1메일 |
| `DROP_REAPPLY_NOTICE` | `drop:{cycleId}:{memberId}` | 주차·회원당 1메일(수동 재발송은 별도 허용) |

> 배치([BAT-J2-A5]) 재실행 시에도 중복 방지 키로 안내 메일이 한 번만 발송된다.

## 5. 비기능 / 운영 고려

| 항목 | 정책 |
|------|------|
| 발송 방식 | 비동기(큐). 트랜잭션 커밋 후 enqueue(아웃박스 패턴 권장) |
| 데이터 정합 | 비즈니스 트랜잭션과 메일 enqueue 분리 — 비즈니스 성공 후 메일 발송 보장 |
| 개인정보 | 본문 내 민감정보 최소화, 발송 로그 보관 기간 정책 적용 |
| 가시성 | 실패/영구 실패 건 관리자 대시보드 노출 + 수동 재발송 |
| 발송 제한 | 인증 메일 재발송 쿨다운([ACC-P2-A2])으로 남용 방지 |
| 첨부/링크 | 인증/재신청 링크는 만료·서명 검증 |
